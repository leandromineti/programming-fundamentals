**File 1: `03-memory/asm-05-data-bss/main.asm` (per ASM-05, D-01, D-02, D-03, D-06, D-07)**

A complete NASM x86-64 program demonstrating `.data` and `.bss` sections. The program:

1. Uses `extern print_uint64` for numeric output
2. Uses `global _start`, sys_write + sys_exit only (per D-01)
3. Demonstrates reading initialized data from `.data` and writing to uninitialized space in `.bss`

The exercise must demonstrate:

**section .data** -- initialized data:
- A 64-bit integer constant using `dq` (e.g., `answer dq 42`)
- A 32-bit integer using `dd` (e.g., `small_val dd 100`)
- A string label using `db` with `equ $ - label` for length (per D-03)

**section .bss** -- uninitialized (zero-filled) data:
- A reserved quadword using `resq` (e.g., `result resq 1`)
- A reserved buffer using `resb` (e.g., `buffer resb 64`)

The program flow:
1. Print header string "--- .data and .bss sections ---\n" via sys_write
2. Load the 64-bit value from `.data` (`mov rax, [answer]`), print via print_uint64
3. Print label "Value from .data (answer):\n" via sys_write (print label BEFORE value, or value after label)
4. Load value from `.data`, add 10 to it, store result in `.bss` (`mov [result], rax`), then reload from `.bss` and print via print_uint64
5. Print label explaining the computation
6. Demonstrate that `.bss` is zero-initialized: load from `result` before writing to it (at a second bss variable), print the zero value
7. Print footer "--- done ---\n"

Concrete output format:
```
--- .data and .bss sections ---
Value from .data:
42
Computed and stored in .bss (42 + 10):
52
.bss is zero-initialized:
0
--- done ---
```

Inline comments must explain:
- `.data` is loaded from the ELF binary -- values are set at compile time
- `.bss` takes no space in the binary -- it's zero-filled at load time
- The difference between `dq` (8 bytes), `dd` (4 bytes), `dw` (2 bytes), `db` (1 byte)
- The difference between `resq` (reserve quadword) and `resb` (reserve bytes)

**CRITICAL: Register preservation around print_uint64 calls.** Use callee-saved registers (rbx, r12-r15) to hold values across calls.

**File 1: `03-memory/asm-06-addressing/main.asm` (per ASM-06, D-01, D-02, D-06, D-07)**

A complete NASM x86-64 program demonstrating three addressing modes. The program:

1. Uses `extern print_uint64` for numeric output
2. Uses `global _start`, sys_write + sys_exit only (per D-01)

Define an array of three 64-bit values in `.data`:
```nasm
section .data
    values  dq  100, 200, 300    ; three 64-bit integers
```

Demonstrate each addressing mode by loading and printing a value:

**Mode 1 -- Direct (label addressing):**
```nasm
    mov     rax, [values]        ; load first element directly by label
    ; rax = 100
```
Print label "Direct [values]:\n" then print 100 via print_uint64.

**Mode 2 -- Register-indirect:**
```nasm
    mov     rbx, values          ; rbx = address of values array
    mov     rax, [rbx]           ; load from address in rbx (first element)
    ; rax = 100
```
Print label "Indirect [rbx]:\n" then print 100 via print_uint64.

**Mode 3 -- Base + displacement:**
```nasm
    mov     rax, [rbx + 8]       ; load second element (8 bytes offset for dq)
    ; rax = 200
    mov     rax, [rbx + 16]      ; load third element (16 bytes offset)
    ; rax = 300
```
Print label "Base+offset [rbx+8]:\n" then print 200, print label "Base+offset [rbx+16]:\n" then print 300.

Concrete output:
```
--- addressing modes ---
Direct [values]:
100
Indirect [rbx]:
100
Base+offset [rbx+8]:
200
Base+offset [rbx+16]:
300
--- done ---
```

Inline comments must explain:
- Direct: the label `values` is resolved to an absolute address by the linker
- Indirect: `mov rbx, values` puts the ADDRESS in rbx; `[rbx]` dereferences it (this IS a pointer)
- Base+displacement: `[rbx + 8]` means "8 bytes after the address in rbx" -- for 64-bit (dq) elements, each element is 8 bytes apart
- Foreshadow: "In C, `arr[1]` compiles to something like `[rbx + 8]` -- this is pointer arithmetic"

**CRITICAL: Register preservation.** Use callee-saved register (rbx) for base address across print_uint64 calls since print_uint64 preserves callee-saved registers.

**File 1: `03-memory/asm-07-array-iteration/main.asm` (per ASM-07, D-01, D-02, D-06, D-07)**

A complete NASM x86-64 program that iterates over an array using indexed addressing with the full SIB (Scale-Index-Base) form. The program:

1. Uses `extern print_uint64` for numeric output
2. Uses `global _start`, sys_write + sys_exit only (per D-01)

Define an array of 64-bit integers in `.data`:
```nasm
section .data
    arr     dq  10, 20, 30, 40, 50    ; array of 5 quadwords
    arr_len equ 5                       ; number of elements (not bytes)
```

The iteration loop uses the full SIB addressing form:
```nasm
    mov     rbx, arr            ; base pointer to array start
    xor     r12, r12            ; index = 0 (callee-saved register)

.loop:
    mov     rdi, [rbx + r12*8]  ; load arr[r12]; scale=8 because each element is 8 bytes (dq)
    call    print_uint64        ; print current element

    inc     r12                 ; index++
    cmp     r12, arr_len        ; compare index with array length
    jl      .loop               ; loop while index < arr_len
```

Output:
```
--- array iteration ---
10
20
30
40
50
--- done ---
```

Inline comments must explain:
- `[rbx + r12*8]` is the SIB (Scale-Index-Base) addressing form
- Scale factor 8 because `dq` elements are 8 bytes each
- This is EXACTLY what C compiles `arr[i]` to: `base + index * sizeof(element)`
- The index register (r12) is callee-saved, so it survives print_uint64 calls
- The base register (rbx) is callee-saved, so it also survives print_uint64 calls
- `arr_len equ 5` is an assembler constant, not a memory variable -- it's substituted at assembly time

**CRITICAL: Both the base (rbx) and index (r12) registers must be callee-saved** to survive print_uint64 calls without push/pop overhead.
